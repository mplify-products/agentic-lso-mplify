Working GroupAgentic LSO Reference Architecture and Blueprints (MP0092)
Introducing Draft definition
What is Agentic LSO?
Agentic LSO · Working Definition
The application of AI agents and the Model Context Protocol (MCP) to Lifecycle Service Orchestration — allowing AI agents acting on behalf of stakeholders to negotiate, quote, order, and manage inter-provider services through standardised, machine-readable interfaces.
Agents augment human operations teams — they do not replace the LSO API programme; they extend its reach to agentic consumers and providers.
Reference Architecture Current draft
What is the Agentic LSO Reference Architecture?
The role MEF 55.1 played for LSO APIs.
The Reference Architecture is being developed to underpin the agentic layer the way MEF 55.1 underpinned the LSO API programme — defining entities, interfaces, and how they relate to stakeholders.
What the current draft specifies.
The types of agents required and which stakeholders they belong to; the types of MCP servers that expose LSO capabilities (one per LSO API domain); the protocols connecting them — A2A, MCP, LSO API — and the role of Legal Identity attached to each entity.
What it does not specify.
Implementation details of any specific agent or MCP server — those belong to the Blueprints (see next slides).
Reference Architecture Proposed
Proposed Agentic LSO Abstract Architecture
Generalised multi-agent topology
The abstract architecture generalises the proof-of-concept into a vendor- and API-neutral view. One MCP Server per LSO API domain; one dedicated sub-agent per MCP Server; a Seller Agent orchestrating them via A2A; and a Buyer Agent negotiating with one or more Seller Agents.
Agent Levels — definitions
Level 1 — Domain-specific agents, each dedicated to a single LSO API via its MCP Server. Stateless tool operators.
Level 2 — Orchestrating agents that coordinate multiple Level 1 agents within a single seller’s domain. Per-seller scope.
Level 3 — Cross-domain agents that negotiate with multiple Level 2 agents across organisational boundaries.

Blueprints · Part 1 of 3 In progress
What are the Agentic LSO Blueprints?
If the Reference Architecture says "a Quote MCP Server exists, and it sits here", a Blueprint says "and this is exactly what tools it exposes, what its identity claims look like, and how it must behave."
A Blueprint is a non-normative specification
Each Blueprint covers one type of entity in the Reference Architecture — for example a Quote MCP Server, or a buyer-side Agent. It specifies the tools, resources, and prompts the entity exposes, the identity model it implements, the expected behaviour, and the conformance criteria.
Blueprint demo — current scope
One Blueprint per MCP Server type in the Reference Architecture (today: Product Catalog, Address Validation, POQ, Quote, Product Order, Product Inventory), plus one or more Agent Blueprints.

Blueprints · Part 2 of 3 Open question
MCP server or agent — where does the logic live?
Two design philosophies are in play. The Blueprints will need to take a reference position on the spectrum between them.
Approach A
Rich MCP server, thin agent
Push instructions, resources, prompts, validation, and business rules into the MCP server. Agents stay general-purpose; servers become opinionated about their domain.
Pros: easier to certify a server's behaviour; centralised domain logic; agent vendors do not need deep LSO knowledge.
Cons: fatter, more complex servers; harder to update per-buyer policy.
Approach B
Minimised scope per entity
Each MCP server and each agent does as little as possible. Logic is distributed; complex flows are composed at the agent layer. Attack surface stays small per component.
Pros: small attack surface; easier to reason about each entity's permissions; easier to swap implementations.
Cons: more orchestration burden on agents; potentially more A2A traffic; harder to enforce consistent domain rules.
Most real deployments will sit somewhere on this spectrum. The Blueprints should be permissive enough to accommodate both — but the reference position is still being chosen. Where should it sit?
Blueprints · Part 3 of 3 Open question
When agents spin up sub-agents, do they use the Blueprint?
A second open governance question. Three positions are on the table — no committee position yet.
01
Yes — always
Any agent in the system — top-level or spawned — must conform to an Agent Blueprint. Guarantees uniform behaviour and identity model end-to-end.
Strongest guarantees. Heaviest implementation tax.
02
No — only top-level agents
Sub-agents are internal implementation. Conformance applies at the boundary where the agent talks to MCP servers or peer agents.
Simpler. Creates an invisible "shadow agent" layer.
03
Conditional — at trust boundaries
Sub-agents conform when they cross a trust boundary (e.g. invoke another organisation's MCP server) but not when purely internal.
Pragmatic. Requires a clear definition of "trust boundary".
Design considerations In progress
Customisation: where, and by whom?
Different operators expose different capabilities, products, and conventions. A key architectural question is: for each of these customisation areas, which component — the agent, the MCP server, or both — should be responsible for carrying and applying that operator-specific context?
User Journey
Which LSO APIs are available at this operator, and which flows they support — determines what the agent can attempt on the buyer's behalf.
Use Cases
From those LSO APIs, which optional capabilities and use cases are enabled — shapes the set of operations the agent can offer to the buyer.
Products
What products are offered and their constraints — governs what the agent can quote, order, and manage on this operator's platform.
Onboarding
Operator-specific conventions: address representations, immediate or deferred response modes, notification patterns, contacts, and similar deployment parameters.
Design considerations In progress
Standard Product Payloads
The AI equivalent of the LSO Blending tool — far simpler. Each standard product ships as a self-contained copy-paste package, ready for immediate use and further customisation.
Replacing the LSO Blending tool
The LSO Blending tool required engineers to manually identify, combine, and validate the right Mplify product specifications. Standard Product Payloads deliver that knowledge pre-packaged — structured for direct AI agent consumption, with no manual merging required.
Content preparation flow
Business Requirements (Std.) → Schema and Dev Guide (Std.) → AI Agent Assets (Guide, Validation, Examples)
For discussion
Is this the right set of components for a standard product package? Are there additional assets — or components to remove?
Package contents · draft blueprint
Registry Entry
Keyword-to-product mapping — lets the agent discover which product applies to a buyer request from natural-language terms.
Knowledge
Enumerated list of all assets with descriptions — the agent's index for discovering what is available and what each asset covers.
AI-Guide
An AI-agent-optimised rewrite of the Mplify standard developer guide — structured for reasoning, not human reading.
Examples
A curated JSON collection of representative payloads covering common and edge-case scenarios for this product.
Validation Rules
All requirements explicitly extracted from the guide into a structured list — the agent's fast-path for compliance checking without re-reading the full guide.
Design considerations Proposed term
Harness Engineering
The implementation discipline that sits between an architectural specification and a working agentic deployment. A harness adapts a general-purpose component — an LLM, an existing API, a test runner — to a specific, constrained, verifiable role in the Agentic LSO stack.
01
Context Harnesses
Pre-packaged bundles of knowledge, instructions, examples, and validation rules that condition a general-purpose AI agent to operate correctly within a specific product domain — without human pre-processing. Standard Product Payloads are the reference implementation.
Shapes the model's understanding.
02
Protocol Harnesses
MCP Servers act as protocol harnesses: they adapt LSO (or non-LSO) APIs into AI-consumable tool surfaces. The agent interacts only with the harness interface; what lies behind it — a legacy system, microservice, or emulator — is an implementation detail.
Shapes the model's reach.
03
Validation Harnesses
Emulators and conformance test suites are validation harnesses: scaffolding that replays known-good flows, asserts Blueprint conformance, and provides the assurance signals needed before connecting to live partners.
Shapes the model's trustworthiness.

For discussion
Can we be 100% sure the agent does exactly what we want?
Unlike code, agent behaviour is probabilistic — the harness constrains it, but does not fully determine it. What level of assurance is achievable, and what does "good enough" look like for inter-provider LSO flows?
Blueprints specify what a conforming harness must do. Harness engineering is how you build one — and the core implementation skill the Agentic LSO programme produces.
Design considerations In progress
Supporting non-LSO APIs
Agentic LSO can wrap pre-existing, non-standard provider APIs — important for members who have not yet implemented LSO APIs (and may not).
Mechanism — current thinking
An MCP Server can front any provider back-end. It does not require the underlying API to be an LSO API. The MCP Server presents an LSO-shaped interface to the agent; what it talks to behind that interface is its own concern.
Implication. Operators with legacy ordering APIs can still expose an LSO-conformant agentic surface — Agentic LSO becomes a migration path, not a prerequisite.
Pros and cons under discussion
Pros
Lowers barrier to entry for Agentic LSO.
Allows incremental migration toward full LSO APIs.
Agent ecosystem is insulated from per-operator variation.
Cons
Each operator wrapping a proprietary back-end is solving the translation problem alone — a fragmentation risk.
Conformance testing is harder without a standardised back-end.
Semantic drift can creep in across operators.
A bridge, not a destination — but that framing is itself open.

Certification In progress
Agentic LSO and Mplify certification
Agentic LSO has a dual relationship with certification. Both directions are in scope; sequencing is open.
As a tool for
Agentic LSO drives certification
Agents and emulators can drive conformance test suites against any LSO implementation — automating what is today a largely manual certification process. Agentic LSO becomes part of the certification machinery.
Faster to deliver.
As a subject of
Agentic LSO gets certified
Conforming agents and MCP servers — built to Blueprints — can themselves be certified. The committee is starting to define what "Agentic LSO Certified" would mean for each Blueprint type.
More strategically important.
Both are being pursued in parallel. Which should the programme prioritise first?

Design considerations Cross-cutting concern
Security — across trust boundaries
Each deployment scenario from slides 14–16 defines a different set of trust boundaries between participants. Security requirements are not uniform — they shift depending on who owns which components and where automated decisions cross organisational borders.
01
Agent identity & legal-entity binding
An agent acting on behalf of a buyer or seller must be cryptographically bound to a recognised legal entity. Trust anchors, certificate chains, and revocation must be defined — especially for A2A (Slide 15) and broker/multi-party topologies (Slide 16) where no human confirms each individual action.
02
Authorisation at MCP & A2A boundaries
Each integration boundary — LSO API, MCP tool call, A2A message — requires its own access-control model. A buyer agent calling a seller MCP server (Slide 15) must be scoped to permitted operations only; third-party chains (Slide 16) raise delegated-authority and transitive-trust questions.
03
Prompt injection & adversarial inputs
LLM-based agents are susceptible to prompt injection through malicious tool responses or crafted A2A messages. Seller or intermediary data returned to a buyer agent — catalogue entries, product descriptions, error messages — could embed instructions that redirect agent behaviour across all topologies.
04
Data minimisation & confidentiality
Agent context windows accumulate sensitive commercial data across tool calls. What data is shared, persisted in agent memory, or logged must be governed — particularly in broker topologies (Slide 16) where an intermediary sees both buyer intent and seller pricing simultaneously.
05
Audit trail & non-repudiation
Automated agents commit to orders, quotes, and SLAs without real-time human sign-off. Immutable, signed logs of agent decisions and the inputs that drove them are required for dispute resolution — and may carry regulatory implications depending on jurisdiction and the deployment topology in use.
06
Blueprint conformance as a security control
Restricting agents to Mplify Blueprint-defined workflows limits the attack surface — a conformant agent cannot be instructed to invoke arbitrary operations. Enforcing Blueprint conformance at every trust boundary (Slides 14–16) turns an interoperability mechanism into a policy-enforcement point.
