What is Agentic LSO?
Agentic LSO · Working Definition
The application of AI agents and the Model Context Protocol (MCP) to Lifecycle Service Orchestration — allowing AI agents acting on behalf of stakeholders to negotiate, quote, order, and manage inter-provider services through standardised, machine-readable interfaces.
Agents augment human operations teams — they do not replace the LSO API programme; they extend its reach to agentic consumers and providers.
Reference Architecture Current draft
What is the Agentic LSO Reference Architecture?
A.
The role MEF 51.1 played for LSO APIs. The Reference Architecture is being developed to underpin the agentic layer the way MEF 51.1 underpinned the LSO API programme — defining entities, interfaces, and how they relate to stakeholders.
B.
What the current draft specifies. The types of agents required and which stakeholders they belong to; the types of MCP servers that expose LSO capabilities (one per LSO API domain); the protocols connecting them — A2A, MCP, LSO API — and the role of Legal Identity attached to each entity.
C.
What it does not specify. Implementation details of any specific agent or MCP server — those belong to the Blueprints (see next three slides).

Blueprints · Part 1 of 3 In progress
What are the Agentic LSO Blueprints?
If the Reference Architecture says "a Quote MCP Server exists, and it sits here", a Blueprint says "and this is exactly what tools it exposes, what its identity claims look like, and how it must behave."
A Blueprint is a non-normative specification
Each Blueprint covers one type of entity in the Reference Architecture — for example a Quote MCP Server, or a buyer-side Agent. It specifies the tools, resources, and prompts the entity exposes, the identity model it implements, the expected behaviour, and the conformance criteria.
Blueprint inventory — current scope
One Blueprint per MCP Server type in the Reference Architecture (today: Product Catalog, Address Validation, POQ, Quote, Product Order, Product Inventory), plus one or more Agent Blueprints.
For discussion: is one Blueprint per MCP server the right granularity, or fewer generic Blueprints with per-domain profiles?
Blueprints · Part 2 of 3 Open question
MCP server or agent — where does the logic live?
Two design philosophies are in play. The Blueprints will need to take a reference position on the spectrum between them.
Approach A
Rich MCP server, thin agent
Push instructions, resources, prompts, validation, and business rules into the MCP server. Agents stay general-purpose; servers become opinionated about their domain.
Pros: easier to certify a server's behaviour; centralised domain logic; agent vendors do not need deep LSO knowledge.
Cons: fatter, more complex servers; harder to update per-buyer policy.
Approach B
Minimised scope per entity
Each MCP server and each agent does as little as possible. Logic is distributed; complex flows are composed at the agent layer. Attack surface stays small per component.
Pros: small attack surface; easier to reason about each entity's permissions; easier to swap implementations.
Cons: more orchestration burden on agents; potentially more A2A traffic; harder to enforce consistent domain rules.
Most real deployments will sit somewhere on this spectrum. The Blueprints should be permissive enough to accommodate both — but the reference position is still being chosen. Where should it sit?
Blueprints · Part 3 of 3 Open question
When agents spin up sub-agents, do they use the Blueprint?
A second open governance question. Three positions are on the table — no committee position yet.
01
Yes — always
Any agent in the system — top-level or spawned — must conform to an Agent Blueprint. Guarantees uniform behaviour and identity model end-to-end.
Strongest guarantees. Heaviest implementation tax.
02
No — only top-level agents
Sub-agents are internal implementation. Conformance applies at the boundary where the agent talks to MCP servers or peer agents.
Simpler. Creates an invisible "shadow agent" layer.
03
Conditional — at trust boundaries
Sub-agents conform when they cross a trust boundary (e.g. invoke another organisation's MCP server) but not when purely internal.
Pragmatic. Requires a clear definition of "trust boundary".
Design considerations In progress
Customisation: where, and by whom?
Blueprints are non-normative as published documents, but contain specific normative elements — and every deployment customises around them. The current direction is to keep those normative elements at the top layer and leave the lower layers explicitly configurable. The exact boundary is open.
Blueprint baseline
The normative elements within the Blueprint — identity model, tool interface, conformance criteria. The reference point against which conformance is assessed.
API-domain specialisation
Variations driven by which LSO API the entity wraps — Quote and Product Inventory have very different access patterns: short-lived offers vs long-lived state.
Use-case specialisation
Variations driven by the service ordered — Carrier Ethernet and IP Transit may need different POQ logic.
Operator policy
Per-deployment configuration: pricing rules, partner whitelists, identity policy, rate limits.
Run-time context
What a specific buyer asks for in a specific session — fully dynamic, never prescribed.
Design considerations In progress
Supporting non-LSO APIs
Agentic LSO can wrap pre-existing, non-standard provider APIs — important for members who have not yet implemented LSO APIs (and may not).
Mechanism — current thinking
An MCP Server can front any provider back-end. It does not require the underlying API to be an LSO API. The MCP Server presents an LSO-shaped interface to the agent; what it talks to behind that interface is its own concern.
Implication. Operators with legacy ordering APIs can still expose an LSO-conformant agentic surface — Agentic LSO becomes a migration path, not a prerequisite.
Pros and cons under discussion
Pros
Lowers barrier to entry for Agentic LSO.
Allows incremental migration toward full LSO APIs.
Agent ecosystem is insulated from per-operator variation.
Cons
Each operator wrapping a proprietary back-end is solving the translation problem alone — a fragmentation risk.
Conformance testing is harder without a standardised back-end.
Semantic drift can creep in across operators.
A bridge, not a destination — but that framing is itself open.

Certification In progress
Agentic LSO and Mplify certification
Agentic LSO has a dual relationship with certification. Both directions are in scope; sequencing is open.
As a tool for
Agentic LSO drives certification
Agents and emulators can drive conformance test suites against any LSO implementation — automating what is today a largely manual certification process. Agentic LSO becomes part of the certification machinery.
Faster to deliver.
As a subject of
Agentic LSO gets certified
Conforming agents and MCP servers — built to Blueprints — can themselves be certified. The committee is starting to define what "Agentic LSO Certified" would mean for each Blueprint type.
More strategically important.
Both are being pursued in parallel. Which should the programme prioritise first?
Design considerations · Part 1 of 2 Open question
Agent deployment scenarios — two-party
Agentic LSO has to work across a range of deployment topologies. The first three describe how a single buyer and single seller might interact — varying only in which side runs an agent.
01
Seller-side agent only
A seller exposes an agent (and MCP servers) to the world; the buyer interacts through traditional means — portal, human operator, or scripted LSO API calls. The agentic surface lives entirely on one side.
02
Buyer-side agent only
A buyer operates an agent that consumes existing LSO APIs (or non-LSO APIs, see Slide 8) directly. The seller side is unchanged. This is often the easiest entry point for member organisations.
03
A2A — buyer and seller agents
Both sides run agents that negotiate over A2A. This is the canonical Agentic LSO topology and the one the Reference Architecture is primarily drawn against.
Design considerations · Part 2 of 2 Open question
Agent deployment scenarios — multi-party
Beyond the two-party cases, Agentic LSO has to support topologies with more than two participants. These raise additional questions about identity, trust boundaries, and concurrency.
04
Third-party agents in the middle
An intermediary agent — broker, aggregator, or marketplace — sits between buyer and seller agents. Raises clear questions about Legal Identity, trust boundaries, and where Blueprint conformance applies.
05
One buyer, many sellers
A buyer agent interacts with multiple seller agents — sequentially for comparison, or simultaneously for parallel quoting. Concurrency, fairness, and disclosure of competing offers all need consideration.
For discussion
Across both two-party and multi-party scenarios — which should the Reference Architecture and Blueprints prioritise? Are there deployment topologies missing from these lists?
What next Candidate list
Use cases — where to focus first
A candidate shortlist for the first Agentic LSO pilots, with brief rationale. The list is open — SMEs are asked to weigh in on prioritisation and additions.
01
Quote-to-Order for Carrier Ethernet
The canonical LSO flow. Well-understood, easy to compare against existing API behaviour.
02
Address validation & serviceability
Narrow scope, low risk, high frequency. A good first agentic touchpoint.
03
Product Inventory queries
Read-only, low blast radius. Good for testing agent identity and authorisation.
04
Cross-operator POQ
Exercises A2A between agents from different operators. Higher coordination cost.
05
Trouble-ticketing & incident triage
High SME interest, but typically requires non-LSO APIs in many deployments (see Slide 8).
06
Multi-domain ordering
Long-term ambition. Unlikely as a first pilot — included for completeness.
